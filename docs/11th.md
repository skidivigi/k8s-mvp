# Одиннадцатый шаг: RBAC

На этом шаге разбираем `RBAC`.

`RBAC` расшифровывается как:

```text
Role-Based Access Control
```

То есть контроль доступа на основе ролей.

RBAC отвечает на вопрос:

```text
Кто может делать какие действия с какими Kubernetes-ресурсами?
```

## Зачем нужен RBAC

В Kubernetes много ресурсов:

```text
pods
deployments
services
secrets
configmaps
persistentvolumeclaims
namespaces
nodes
```

И не каждому пользователю или приложению нужно давать полный доступ ко всему кластеру.

Например:

```text
backend-приложению можно читать ConfigMap;
CI/CD системе можно обновлять Deployment;
monitoring-agent может читать Pod и Node;
обычный dev может смотреть Pod, но не удалять namespace;
приложению нельзя читать все Secrets в кластере.
```

RBAC позволяет описать эти правила декларативно.

## Основные сущности RBAC

В RBAC есть несколько ключевых объектов:

```text
ServiceAccount
Role
ClusterRole
RoleBinding
ClusterRoleBinding
```

## ServiceAccount

`ServiceAccount` — это identity для Pod внутри Kubernetes.

Обычный пользователь работает через kubeconfig.

А приложение внутри Pod работает через ServiceAccount.

Схема:

```text
Pod
  ↓
ServiceAccount
  ↓
права через RoleBinding / ClusterRoleBinding
```

Если Pod не указал ServiceAccount явно, Kubernetes использует default ServiceAccount в namespace.

В нашем примере создаем отдельный ServiceAccount:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: demo-reader
  namespace: lab
```

## Role

`Role` описывает права внутри одного namespace.

Пример:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: lab
rules:
  - apiGroups:
      - ""
    resources:
      - pods
    verbs:
      - get
      - list
      - watch
```

Эта Role говорит:

```text
В namespace lab можно читать pods.
```

Но сама по себе Role никому не выдает права.

Она просто описывает набор разрешений.

## RoleBinding

`RoleBinding` связывает пользователя, группу или ServiceAccount с Role.

Пример:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: demo-reader-pod-reader
  namespace: lab
subjects:
  - kind: ServiceAccount
    name: demo-reader
    namespace: lab
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

Это значит:

```text
ServiceAccount demo-reader получает права из Role pod-reader в namespace lab.
```

Схема:

```text
ServiceAccount demo-reader
  ↓ RoleBinding
Role pod-reader
  ↓ rules
можно get/list/watch pods в namespace lab
```

## ClusterRole

`ClusterRole` похож на Role, но не привязан к одному namespace.

Его используют для:

```text
кластерных ресурсов;
одинаковых прав в разных namespace;
ресурсов уровня всего кластера.
```

Примеры cluster-wide ресурсов:

```text
nodes
namespaces
persistentvolumes
clusterroles
storageclasses
```

Например, `nodes` нельзя читать через обычную namespaced Role, потому что `nodes` — ресурс уровня кластера.

## ClusterRoleBinding

`ClusterRoleBinding` связывает subject с ClusterRole на уровне всего кластера.

Это мощная штука.

Если дать через ClusterRoleBinding много прав, они будут работать во всем кластере.

С этим нужно быть осторожным.

## Role vs ClusterRole

| Объект             | Где действует                            |
| ------------------ | ---------------------------------------- |
| Role               | Внутри одного namespace                  |
| ClusterRole        | На уровне кластера или как шаблон прав   |
| RoleBinding        | Выдает Role/ClusterRole внутри namespace |
| ClusterRoleBinding | Выдает ClusterRole на весь кластер       |

Важный нюанс:

`RoleBinding` может ссылаться не только на `Role`, но и на `ClusterRole`.

Тогда права из `ClusterRole` будут применены только в namespace этого `RoleBinding`.

Схема:

```text
ClusterRole pod-reader
  ↓
RoleBinding в namespace lab
  ↓
права работают только в lab
```

А вот:

```text
ClusterRole pod-reader
  ↓
ClusterRoleBinding
  ↓
права работают во всем кластере
```

## Verbs

`verbs` — это действия, которые разрешены.

Частые verbs:

```text
get
list
watch
create
update
patch
delete
deletecollection
```

Пример:

```yaml
verbs:
  - get
  - list
  - watch
```

Это read-only доступ.

Для изменения ресурсов нужны:

```text
create
update
patch
delete
```

## Resources

`resources` — это Kubernetes-ресурсы, к которым применяются права.

Например:

```yaml
resources:
  - pods
```

Можно указать несколько:

```yaml
resources:
  - pods
  - services
  - configmaps
```

## apiGroups

`apiGroups` указывает API-группу ресурса.

Для базовых ресурсов используется пустая строка:

```yaml
apiGroups:
  - ""
```

К базовой API-группе относятся, например:

```text
pods
services
configmaps
secrets
persistentvolumeclaims
```

Для Deployment нужна API-группа:

```yaml
apiGroups:
  - apps
resources:
  - deployments
```

Потому что Deployment имеет:

```yaml
apiVersion: apps/v1
kind: Deployment
```

Для RBAC-ресурсов API-группа:

```yaml
apiGroups:
  - rbac.authorization.k8s.io
```

## Что создаем в этом уроке

Создаем ServiceAccount:

```text
demo-reader
```

Создаем Role:

```text
pod-reader
```

Эта Role разрешает:

```text
get/list/watch pods
```

Создаем RoleBinding:

```text
demo-reader-pod-reader
```

Он выдает ServiceAccount `demo-reader` права из Role `pod-reader`.

Потом запускаем Pod:

```text
rbac-test-pod
```

Этот Pod работает от ServiceAccount:

```text
demo-reader
```

```bash

```

Это нормально.

Это значит, что RBAC работает.

## 7. Почему Pod может обращаться к Kubernetes API

Когда Pod запускается с ServiceAccount, Kubernetes монтирует внутрь Pod специальные файлы:

```text
/var/run/secrets/kubernetes.io/serviceaccount/token
/var/run/secrets/kubernetes.io/serviceaccount/ca.crt
/var/run/secrets/kubernetes.io/serviceaccount/namespace
```

`kubectl` внутри Pod использует этот token, чтобы обратиться к Kubernetes API.

Проверить можно так:

```bash
kubectl exec -it rbac-test-pod -n lab -- ls -la /var/run/secrets/kubernetes.io/serviceaccount
```

Там будут файлы token, certificate и namespace.

## 8. default ServiceAccount

Если в Pod не указать:

```yaml
serviceAccountName:
```

то Kubernetes использует ServiceAccount:

```text
default
```

из того же namespace.

То есть вот такой Pod:

```yaml
spec:
  containers:
    - name: app
      image: nginx
```

фактически будет запущен с:

```yaml
serviceAccountName: default
```

Но default ServiceAccount обычно не должен иметь лишних прав.

Production-подход:

```text
создавать отдельный ServiceAccount под приложение;
давать ему минимальные нужные права;
не использовать cluster-admin без необходимости;
не выдавать права на все secrets;
разделять права по namespace.
```

## 9. Least Privilege

Главный принцип RBAC:

```text
Давать только те права, которые реально нужны.
```

Это называется:

```text
least privilege
```

Например, если приложению нужно только читать ConfigMap, не надо давать ему:

```text
delete pods
list secrets
update deployments
cluster-admin
```

Плохой пример:

```yaml
rules:
  - apiGroups:
      - "*"
    resources:
      - "*"
    verbs:
      - "*"
```

Это почти полный доступ.

Такое можно использовать только в очень редких случаях и осознанно.

## 10. RoleBinding на ClusterRole

Иногда удобно создать `ClusterRole`, но выдать его только в одном namespace через `RoleBinding`.

Например:

```text
ClusterRole view
  ↓
RoleBinding в namespace lab
  ↓
права view только в lab
```

Это часто используется, потому что в Kubernetes уже есть стандартные ClusterRole:

```text
view
edit
admin
cluster-admin
```

Пример:

```bash
kubectl create rolebinding demo-view \
  --clusterrole=view \
  --serviceaccount=lab:demo-reader \
  -n lab
```

Это выдаст ServiceAccount `demo-reader` права `view` только в namespace `lab`.

## 11. Когда нужен ClusterRoleBinding

`ClusterRoleBinding` нужен, когда права должны действовать на весь кластер.

Например:

```text
monitoring-agent читает nodes;
ingress-controller смотрит ingress во всех namespace;
cert-manager работает с certificate resources;
CNI plugin управляет network resources;
storage controller управляет PV.
```

Но ClusterRoleBinding нужно использовать аккуратно.

Он шире, чем RoleBinding.

## Что важно понять после RBAC

После этого шага нужно зафиксировать:

```text
RBAC управляет доступом к Kubernetes API.

ServiceAccount — identity для Pod.

Role описывает права внутри namespace.

ClusterRole описывает права уровня кластера или reusable-набор прав.

RoleBinding выдает Role/ClusterRole внутри namespace.

ClusterRoleBinding выдает ClusterRole на весь кластер.

verbs — действия: get, list, watch, create, update, patch, delete.

resources — Kubernetes-ресурсы: pods, services, deployments, secrets.

apiGroups зависят от apiVersion ресурса.

Минимальные права лучше, чем широкие права.

kubectl auth can-i помогает проверять доступ.
```

## Мини-конспект

```text
ServiceAccount
  ↓
RoleBinding
  ↓
Role
  ↓
rules
  ↓
что можно делать в Kubernetes API
```

Пример нашего урока:

```text
demo-reader
  ↓
demo-reader-pod-reader
  ↓
pod-reader
  ↓
можно get/list/watch pods в namespace lab
```

Главная мысль:

```text
Role описывает права.
RoleBinding выдает эти права.
ServiceAccount использует эти права внутри Pod.
```
