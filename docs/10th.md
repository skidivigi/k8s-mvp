# Десятый шаг: Metrics Server и kubectl top

На этом шаге добавляем `Metrics Server`.

`Metrics Server` нужен, чтобы Kubernetes мог получать текущие метрики потребления CPU и RAM по Node и Pod.

После установки Metrics Server начинают работать команды:

```bash
kubectl top nodes
kubectl top pods
```

## Metrics Server — это не Prometheus

Важно сразу разделить:

```text
Metrics Server ≠ Prometheus
```

`Metrics Server` нужен Kubernetes для базовых resource metrics:

```text
CPU usage
Memory usage
```

Эти данные используют:

```text
kubectl top
Horizontal Pod Autoscaler
Kubernetes Metrics API
```

Prometheus нужен для другого:

```text
долговременное хранение метрик
графики
алерты
метрики приложений
метрики баз данных
метрики брокеров
метрики ingress-controller
```

Коротко:

```text
Metrics Server — для Kubernetes control plane.
Prometheus — для полноценного мониторинга.
```

## Общая схема

```text
kubelet на каждой node
  ↓
Metrics Server
  ↓
Kubernetes Metrics API
  ↓
kubectl top / HPA
```

`Metrics Server` забирает метрики с kubelet на каждой node.

После этого Kubernetes API начинает отдавать ресурсные метрики через API:

```text
/apis/metrics.k8s.io/v1beta1
```

## Что ставим

Создаем файл:

```text
k8s/09-metrics-server/patch.yaml
```

Содержимое:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: metrics-server
  namespace: kube-system
spec:
  template:
    spec:
      containers:
        - name: metrics-server
          args:
            - --cert-dir=/tmp
            - --secure-port=10250
            - --kubelet-preferred-address-types=InternalIP,ExternalIP,Hostname
            - --kubelet-use-node-status-port
            - --metric-resolution=15s
            - --kubelet-insecure-tls
```

Главный параметр для локального kind-кластера:

```text
--kubelet-insecure-tls
```

Он нужен, потому что в локальном kind-кластере Metrics Server может не доверять kubelet TLS certificate.

Для production лучше настраивать нормальные kubelet certificates, а не отключать TLS-проверку.

## Makefile

```makefile
# 10th lab - Metrics Server

.PHONY: install-metrics-server-10
install-metrics-server-10:
	kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

.PHONY: patch-metrics-server-10
patch-metrics-server-10:
	kubectl patch deployment metrics-server -n kube-system --type='merge' --patch-file k8s/09-metrics-server/patch.yaml

.PHONY: rollout-metrics-server-10
rollout-metrics-server-10:
	kubectl rollout status deployment/metrics-server -n kube-system --timeout=180s

.PHONY: setup-metrics-server-10
setup-metrics-server-10:
	kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
	kubectl patch deployment metrics-server -n kube-system --type='merge' --patch-file k8s/09-metrics-server/patch.yaml
	kubectl rollout status deployment/metrics-server -n kube-system --timeout=180s

.PHONY: get-metrics-server-10
get-metrics-server-10:
	kubectl get pods -n kube-system -l k8s-app=metrics-server -o wide

.PHONY: describe-metrics-server-10
describe-metrics-server-10:
	kubectl describe deployment metrics-server -n kube-system

.PHONY: logs-metrics-server-10
logs-metrics-server-10:
	kubectl logs -n kube-system -l k8s-app=metrics-server

.PHONY: top-pods-10
top-pods-10:
	kubectl top pods -n lab

.PHONY: top-pods-all-10
top-pods-all-10:
	kubectl top pods -A

.PHONY: top-nodes-10
top-nodes-10:
	kubectl top nodes

.PHONY: raw-node-metrics-10
raw-node-metrics-10:
	kubectl get --raw "/apis/metrics.k8s.io/v1beta1/nodes" | jq .

.PHONY: raw-pod-metrics-10
raw-pod-metrics-10:
	kubectl get --raw "/apis/metrics.k8s.io/v1beta1/pods" | jq .

.PHONY: delete-metrics-server-10
delete-metrics-server-10:
	kubectl delete -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
```

Если `jq` не установлен, команды `raw-node-metrics-10` и `raw-pod-metrics-10` можно выполнить без:

```bash
| jq .
```

Можно посмотреть сырые данные через Kubernetes API:

```bash

```

```bash

```

```bash

```

Это показывает, что `kubectl top` — это просто удобная команда поверх Metrics API.

## Что такое CPU в kubectl top

В выводе можно увидеть:

```text
CPU(cores)
100m
250m
1
```

Значения:

```text
100m  = 0.1 CPU core
500m  = 0.5 CPU core
1000m = 1 CPU core
```

То есть если Pod потребляет:

```text
25m
```

это примерно:

```text
0.025 CPU core
```

## Что такое memory в kubectl top

Memory показывается как текущее потребление RAM:

```text
MEMORY(bytes)
64Mi
128Mi
1Gi
```

Примеры:

```text
64Mi  = 64 mebibytes
128Mi = 128 mebibytes
1Gi   = 1 gibibyte
```

## Metrics Server и requests/limits

В прошлом шаге мы задавали:

```yaml
resources:
  requests:
    cpu: "100m"
    memory: "64Mi"
  limits:
    cpu: "500m"
    memory: "128Mi"
```

`kubectl top` показывает фактическое текущее потребление.

То есть можно сравнить:

```text
requests — сколько попросили для scheduler
limits   — максимум, который разрешили
top      — сколько реально сейчас используется
```

Пример:

```text
request CPU: 100m
limit CPU:   500m
top CPU:      12m
```

Это значит, что приложение сейчас потребляет меньше, чем запросило.

## Metrics Server и HPA

`Metrics Server` нужен для Horizontal Pod Autoscaler.

HPA может смотреть на CPU/RAM и менять количество реплик.

Схема:

```text
Metrics Server
  ↓
HPA
  ↓
Deployment replicas
```

Например:

```text
CPU вырос
  ↓
HPA увеличил replicas с 2 до 5
```

HPA мы можем разобрать позже отдельным коротким шагом после resources, либо в блоке autoscaling.

## Troubleshooting

### Ошибка: Metrics API not available

Пример:

```text
error: Metrics API not available
```

Проверить:

```bash
make get-metrics-server-10
make logs-metrics-server-10
```

Частые причины:

```text
Metrics Server еще не стартовал.
Metrics Server не может достучаться до kubelet.
Не применился patch с --kubelet-insecure-tls.
Нужно немного подождать после установки.
```

### Ошибка TLS к kubelet

Если в логах видно что-то про certificate или TLS, проверь patch:

```bash
make describe-metrics-server-10
```

В args должен быть:

```text
--kubelet-insecure-tls
```

### kubectl top pods пустой или не сразу работает

После установки нужно немного времени, чтобы Metrics Server собрал первые метрики.

Проверь еще раз:

```bash
make top-pods-all-10
```

## Production-нюанс

В kind мы используем:

```text
--kubelet-insecure-tls
```

Это нормально для учебной локальной среды.

В production так лучше не делать.

В production правильнее:

```text
настроить корректные kubelet serving certificates;
использовать managed add-on в облаке, если он есть;
устанавливать Metrics Server через Helm/GitOps;
мониторить сам Metrics Server.
```

## Что важно понять после десятого шага

После этого шага нужно зафиксировать:

```text
Metrics Server собирает текущие CPU/RAM метрики.

kubectl top берет данные из Metrics Server.

Metrics Server реализует metrics.k8s.io API.

Metrics Server нужен для HPA.

Metrics Server не заменяет Prometheus.

Prometheus нужен для долгой истории, алертов и dashboards.

В kind часто нужен --kubelet-insecure-tls.

requests/limits — это настройки Pod.

kubectl top — это фактическое текущее потребление.
```
